## Conflicting Requirements

- REQ-0001: GitHub Projects v2 作为主面板（视图层）

## Conflict Points

- 面板主事实源冲突：REQ-0001 规定 Projects 为主面板（视图层、可全量重建），REQ-0002 要求 n8n 为主面板并禁止 Projects。
- 治理与权限冲突：n8n 工作流编辑权限与审计策略需要额外加固，否则与 Team OS 的“可追溯/闸门”要求冲突。
- 影响范围：workstream 维度过滤/Roadmap 视图在 Projects 中具备原生能力；若迁移到 n8n，需要重新实现并保证可重建。

## Suggested Options (NEED_PM_DECISION)

### Option A: 维持 Projects 为主面板，n8n 仅做通知/自动化

- Pros:
  - 符合“Projects 视图层，可从真相源全量重建”的原则
  - 降低权限与运维复杂度（n8n 不承担主计划）
- Cons:
  - REQ-0002 的主诉求被否决或需要改写
- Needs PM Decision:
  - 明确 REQ-0002 的状态（DEPRECATED 或降级为“通知补充”）

### Option B: 迁移到 n8n 作为主面板

- Pros:
  - 满足 REQ-0002 的诉求
- Cons:
  - 需要重新定义事实源/回填/审计，且更易引入安全风险
  - Roadmap/Board/Table 视图能力需要自建或依赖第三方
- Needs PM Decision:
  - 是否接受更高的安全与运维成本
  - 迁移计划与回滚策略

### Option C: 分域策略（Projects 作为计划面板；n8n 仅在特定域/通知场景使用）

- Pros:
  - 兼顾部分诉求，避免一刀切
- Cons:
  - 边界需要严格定义，否则会产生“两个真相源”
- Needs PM Decision:
  - 明确哪些数据/视图在 Projects，哪些在 n8n
  - 明确重同步与审计规则
